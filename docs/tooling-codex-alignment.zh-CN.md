# 工具对齐 Codex 的开发验收标准

这份文档用于约束当前仓库里的工具系统改造。目标不是增加更多工具，而是把现有工具收敛成更少、更强、更清晰的能力族，并确保它们的行为、描述、结果结构和调度方式都接近 Codex。

## 总目标

- 默认工具暴露集保持小而高杠杆。
- 模型优先走 `search_workspace -> read_file -> edit_file / exec_command` 这条主链。
- 多文件读取通过多个独立 `read_file` 调用并行完成，而不是通过一个批量读取工具内部串行完成。
- 工具结果必须同时满足“人能看懂”和“模型能继续推理”。

## 当前工具原则

- 一个能力族只保留一个稳定概念。
- 旧的、重叠的、会诱导错误调用的工具优先删除，而不是长期兼容。
- 工具描述必须明确告诉模型什么时候用、什么时候不要用、下一步怎么接。
- 结构化结果是事实源，展示文本只是投影。

## 现有主链

### `search_workspace`

职责：

- 先定位候选文件或文本命中。
- 返回能直接指导下一步 `read_file` 的结构化结果。

验收标准：

- `files` 模式返回的命中必须包含排序信号。
- `text` 模式必须对结果做相关性排序，不能只按扫描顺序返回。
- 结果必须包含足够信息，让模型能判断下一步该读哪个文件。
- 工具描述必须明确说明：它的目标是让下一次 `read_file` 调用变得明显。

通过标准：

- 命中结果中能看到 `score`、`match_kind`，必要时有 `indices`。
- 文本命中会优先把定义式、文件名强命中、路径强命中排到前面。

### `read_file`

职责：

- 单文件精读。
- 返回带行号、截断信息和续读信息的结构化事实。
- 这是 repo 精读工具，不是 fs 原语；底层文件系统读写原语单独由 `read_file_bytes` / `write_file_bytes` 承担。

验收标准：

- 只保留单文件语义。
- 多文件场景必须由多个独立 `read_file` 调用完成。
- 工具描述必须明确告诉模型：多个文件要分开读，并利用运行时并行。
- 截断时必须明确提示下一次从哪里续读。
- 结果必须保留 `next_start_line`。

通过标准：

- `read_file` 的结构化结果里只有一个文件的读事实。
- 截断提示里能直接看到 `next_start_line`。
- `edit_file` 可以稳定把 `read_file` 作为 witness。

### `edit_file`

职责：

- 做精确替换式编辑。
- 依赖之前的 `read_file` witness。

验收标准：

- 不能把 patch 文本、行号前缀、猜测性文本当成合法输入。
- 必须校验 witness 仍然有效。
- 必须在旧内容和新内容之间提供最小但唯一的匹配范围。

通过标准：

- 旧文本被改动后，编辑会拒绝并要求重新读取。
- 多匹配时必须要求更具体的上下文或明确 `replace_all`。

### `exec_command`

职责：

- 只用于 build、test、git、运行时验证和交互式命令会话。

验收标准：

- 不得作为文件编辑的主路径。
- 不得作为 repo 搜索的主路径。
- 必须保留 session 复用和清晰的执行状态。

通过标准：

- 代码里对文件修改命令有明确拒绝或引导。
- 输出能让模型判断命令是否还在运行、是否完成、是否失败。

## 已删除或应删除的旧工具

以下工具不应作为默认工具面的一部分：

- `read_files`
- `list_directory`

原则：

- 只要它们会诱导模型绕开主链，就不保留默认入口。
- 只有在确实能减少步骤且不会制造歧义时，才考虑重新引入。

## 默认工具面验收

默认暴露集只保留高杠杆工具，不应让模型先学会“逛目录”。

通过标准：

- 默认排序能把 `search_workspace` 和 `read_file` 放在前面。
- `list_directory` 不出现在默认主链里。
- 不再保留 `read_files` 的默认调用路径。

## 文档删除门禁

这份文档不能删，除非同时满足下面条件：

1. 所有主链工具的实现已完成并通过测试。
2. 所有相关描述、结构化结果、上下文压缩逻辑都已同步。
3. 默认工具面已经去掉旧工具入口。
4. `cargo test -p agent-tools`、`cargo test -p agent-core`、`cargo test -p cli --no-run` 全部通过。
5. 人工抽查一轮真实对话，确认模型默认会走正确工具序列。

如果任一条件不满足，这份文档必须保留，作为继续开发和验收的基准。

## 开发顺序建议

1. 先删旧工具入口。
2. 再补齐主链工具的 structured result。
3. 最后只做必要的描述和排序收敛。
